# 포트폴리오 사이트 아키텍처 개선 계획 (SPA -> Universal Rendering)

## 1. 현재 구조의 문제점 (SPA & Hash Routing)
*   **구조:** 하나의 주소(`/`)에서 모든 상태(`activeTab`)를 자바스크립트로 관리하며 탭 내용만 화면에서 교체하는 완벽한 SPA(Single Page Application) 형태.
*   **문제점:** AI 크롤러나 검색 엔진 봇(Bot)은 자바스크립트를 완벽하게 실행하거나 탭을 클릭하지 못함. 결과적으로 처음 접속 시 보여지는 '홈' 탭의 내용만 수집하고, 다른 탭(이력서, 자기소개 등)의 내용은 누락됨.
*   **임시 조치:** 모든 탭의 내용을 미리 렌더링하고 비활성 탭은 CSS(`display: none`)로 숨겨두는 꼼수를 사용했으나, 이는 근본적인 해결책이 아니며 페이지 단위의 독립적인 URL이 없어 특정 탭을 타인에게 링크로 공유하기 어려움 (해시 `#resume` 로는 공유 가능하지만 SEO에는 불리함).

## 2. 목표 구조: 유니버설 렌더링 (Isomorphic App)
*   **목표:** 각 탭이 고유한 URL 경로(예: `/resume`, `/introduction`)를 갖도록 설계하여 **MPA(Multi-Page Application)의 SEO 이점**과 **SPA의 부드러운 사용자 경험(UX)**을 모두 취하는 것.
*   **핵심 기술 (Next.js App Router 기반):**
    1.  **서버 사이드 렌더링 (SSR/SSG):** 크롤러나 사용자가 특정 주소(예: `/introduction`)로 처음 접속하면, 서버(빌드 결과물)는 딱 **해당 페이지(자기소개)의 HTML만 즉시(MPA처럼)** 응답함. 크롤러는 불필요한 데이터를 받지 않고 해당 문서의 내용만 완벽하게 수집 가능.
    2.  **클라이언트 사이드 라우팅 (CSR):** 이후 브라우저에서 자바스크립트가 로드되면, 앱은 SPA 모드로 전환됨(Hydration).
    3.  **프리패칭 (Prefetching):** Next.js의 `<Link>` 컴포넌트를 사용하면, 현재 화면에 보이는 링크(예: 다음 탭 버튼)의 대상 페이지(`/career`) 데이터를 **사용자가 클릭하기 전에 백그라운드에서 미리 로드(캐싱)** 해둠.
    4.  **부드러운 전환:** 사용자가 탭을 클릭하면 캐싱된 데이터를 이용해 깜빡임 없이 즉각적으로 화면만 교체됨 (기존 SPA의 장점 유지).

## 3. 리팩토링 단계 (Action Plan)

### 단계 1: 공통 레이아웃 분리 (`app/layout.tsx`)
*   현재 `app/page.tsx`에 있는 전체 껍데기 UI(노트북 스프링, 왼쪽 사이드바, 프로필, 탭 스트립)를 모든 페이지가 공유하도록 `src/app/layout.tsx` (또는 `template.tsx`)로 이동시킵니다.
*   가운데 종이 영역만 `children`으로 받아서 각 페이지 내용이 렌더링되게 합니다.

### 단계 2: 페이지 분리 및 동적 라우팅 설정 (`app/[tab]/page.tsx`)
*   Next.js App Router의 폴더 구조를 활용하여 각 탭별 주소를 생성합니다.
*   `src/app/page.tsx`: 기존의 홈(`home`) 콘텐츠만 표시.
*   `src/app/[tabId]/page.tsx`: 이력서, 자기소개, 경력, About 등 나머지 콘텐츠를 동적으로 렌더링. `generateStaticParams`를 사용하여 빌드 시점에 모든 탭 경로(`/resume`, `/introduction` 등)의 정적 HTML 파일을 생성(SSG)하도록 설정.

### 단계 3: 상태 기반 라우팅에서 URL 기반 라우팅으로 전환
*   **탭 버튼 교체:** `TabButton.tsx` 내의 `<button>` 태그를 Next.js의 `<Link href="/[tabId]">` 로 교체합니다. 이로써 **자동 프리패칭** 기능이 활성화됩니다.
*   **상태 관리 제거:** 기존의 `useTabWheel` 훅에서 관리하던 `activeTab` 상태 대신, Next.js의 `usePathname()` 훅을 사용하여 현재 활성화된 탭(URL 경로)을 파악하고 색상 등을 변경합니다.
*   **해시 동기화 제거:** URL 자체가 바뀌므로 더 이상 해시(`#resume`)를 동기화하는 `useHashSync` 훅은 필요하지 않습니다.

### 단계 4: 마우스 휠 스크롤 전환 (Wheel Navigation) 유지
*   마우스 휠로 탭을 넘기는 기존의 매력적인 UX는 유지해야 합니다.
*   `useTabWheel` 훅을 수정하여, 상태 업데이트(`setActiveTab`) 대신 Next.js의 `useRouter().push('/다음경로')`를 호출하도록 변경합니다.
*   프리패칭 덕분에 휠 스크롤 시에도 화면 전환이 즉각적으로 이루어집니다.

### 단계 5: 잔재 코드 정리 (Cleanup)
*   임시 조치로 사용했던 `display: none` 전체 렌더링 로직을 제거하고, 해당 경로에 맞는 하나의 콘텐츠 컴포넌트만 렌더링하도록 롤백합니다.

## 4. 기대 효과
*   **완벽한 SEO 최적화:** AI 크롤러와 검색 엔진이 각 탭의 내용을 독립적인 문서로 완벽하게 수집하고 색인(Index)할 수 있습니다.
*   **고유 URL 제공:** 특정 탭(예: 자기소개서만)을 타인에게 카카오톡이나 이메일 링크로 깔끔하게 공유할 수 있습니다 (`/introduction`).
*   **초기 로딩 속도 향상:** 처음 접속 시 불필요한 다른 탭의 내용(HTML)까지 한꺼번에 다운로드하지 않아도 되므로 초기 화면 진입이 미세하게 더 빨라집니다.
*   **최상의 UX:** 첫 페이지 이후의 이동은 SPA 방식과 프리패칭의 조화로 인해 기존과 동일하게 깜빡임 없이 즉각적이고 부드러운 노트북 애니메이션 경험을 제공합니다.
